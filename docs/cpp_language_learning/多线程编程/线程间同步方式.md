## 总览

线程同步的本质是两件事：

- 互斥访问共享资源
- 协调线程执行顺序

下面是常见机制的快速比较：

| 机制 | 解决问题 | 典型场景 |
| --- | --- | --- |
| mutex | 临界区互斥 | 多线程改同一对象 |
| condition_variable | 等待条件成立 | 生产者消费者 |
| atomic | 无锁原子读写 | 计数器、标志位 |
| shared_mutex | 多读单写 | 配置缓存、字典查询 |
| semaphore | 限流与许可数控制 | 连接池、任务并发上限 |
| latch/barrier | 多线程阶段同步 | 并行阶段计算 |

---

## mutex

mutex 保护临界区，同一时刻只有一个线程进入。

推荐搭配 lock_guard 或 unique_lock，避免手写 lock/unlock。

---

## condition_variable

用于等待条件变化，不做忙等。典型模型：

- 消费者等待队列非空
- 生产者入队后通知

等待必须写成谓词形式：

```cpp
cv.wait(lock, [] { return ready; });
```

---

## atomic

atomic 适合简单共享状态：

- 原子计数
- 终止标志
- 轻量统计

涉及多个变量的一致性时，atomic 往往不够，需要锁或更复杂并发结构。

---

## shared_mutex

读多写少场景优先考虑：

- 读：shared_lock
- 写：unique_lock

可提升读吞吐，但要注意写饥饿风险与锁升级复杂度。

---

## semaphore（C++20）

信号量控制“同时允许多少线程进入”。

典型场景：

- 同时最多 N 个任务访问数据库。
- 限制线程池中的某类任务并发度。

---

## latch 与 barrier（C++20）

- latch：一次性计数门闩，适合“等一批初始化完成”。
- barrier：可重复阶段栅栏，适合迭代式并行计算。

它们偏向阶段协作，不替代互斥锁。

---

## 如何选择

按需求选最小工具：

1. 只保护一个临界区：mutex。
2. 需要等待事件发生：condition_variable。
3. 仅单变量原子操作：atomic。
4. 读多写少：shared_mutex。
5. 需要并发许可数量：semaphore。
6. 需要阶段同步：latch/barrier。

实践原则：先保证正确，再关注性能。

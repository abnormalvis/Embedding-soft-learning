## 条件变量解决什么问题

条件变量用于线程间协作：

- 一个线程等待某个条件成立。
- 另一个线程修改共享状态后发出通知。

相比轮询检查，条件变量可以让线程阻塞等待，减少 CPU 空转。

---

## 核心 API

头文件：

```cpp
#include <condition_variable>
```

常用接口：

- wait(lock)
- wait(lock, predicate)
- wait_for(lock, duration)
- notify_one()
- notify_all()

条件变量通常与以下对象一起出现：

- std::mutex
- std::unique_lock<std::mutex>
- 共享状态变量（如队列是否为空）

---

## 正确等待模式

必须使用谓词版本等待，防止虚假唤醒。

```cpp
std::unique_lock<std::mutex> lock(m);
cv.wait(lock, [] { return ready; });
```

等价的经典写法：

```cpp
std::unique_lock<std::mutex> lock(m);
while (!ready) {
    cv.wait(lock);
}
```

注意：

- wait 期间会释放锁，被唤醒后会重新加锁。
- 通知不等于条件成立，必须再次检查谓词。

## 生产者消费者示例

```cpp
#include <condition_variable>
#include <iostream>
#include <mutex>
#include <queue>
#include <thread>

std::mutex m;
std::condition_variable cv;
std::queue<int> q;
bool done = false;

void producer() {
    for (int i = 1; i <= 5; ++i) {
        {
            std::lock_guard<std::mutex> lock(m);
            q.push(i);
        }
        cv.notify_one();
    }

    {
        std::lock_guard<std::mutex> lock(m);
        done = true;
    }
    cv.notify_all();
}

void consumer() {
    while (true) {
        std::unique_lock<std::mutex> lock(m);
        cv.wait(lock, [] { return !q.empty() || done; });

        if (!q.empty()) {
            int value = q.front();
            q.pop();
            lock.unlock();
            std::cout << "consume " << value << "\n";
        } else if (done) {
            break;
        }
    }
}

int main() {
    std::thread p(producer);
    std::thread c(consumer);
    p.join();
    c.join();
    return 0;
}
```

编译：

```bash
g++ -std=c++20 -Wall -Wextra -O2 main.cpp -pthread -o app
```

## wait_for 与超时

当你不希望无限期等待时，使用 wait_for：

```cpp
std::unique_lock<std::mutex> lock(m);
bool ok = cv.wait_for(lock, std::chrono::seconds(1), [] { return ready; });
if (!ok) {
    // 超时处理
}
```

适用于网络、设备、任务调度等需要超时回退的场景。

---

## notify_one 与 notify_all

- notify_one：唤醒一个等待线程，适合单任务消费。
- notify_all：唤醒所有等待线程，适合状态广播。

经验：

- 单生产者单消费者常用 notify_one。
- 状态切换（如 stop/done）常用 notify_all。

---

## 常见坑

1. 没有谓词等待，导致虚假唤醒后读到错误状态。
2. 修改共享状态不加锁，导致通知丢失或可见性问题。
3. 在持锁状态执行耗时逻辑，导致其他线程长期饥饿。
4. 只通知不改状态，等待线程醒来后仍继续阻塞。
5. 把 sleep_for 当同步原语使用，造成不稳定时序。

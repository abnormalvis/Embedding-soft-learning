## 读写锁是什么

读写锁允许：

- 多个读线程同时访问（共享锁）
- 写线程独占访问（独占锁）

在“读多写少”场景下，它通常比普通 mutex 有更好吞吐。

---

## shared_mutex 基本模型

C++17 提供 shared_mutex：

- 读：shared_lock<shared_mutex>
- 写：unique_lock<shared_mutex>

头文件：

```cpp
#include <shared_mutex>
```

## 读锁与写锁示例

```cpp
#include <iostream>
#include <shared_mutex>
#include <thread>
#include <vector>

std::shared_mutex rw;
int data_value = 0;

void reader(int id) {
    std::shared_lock<std::shared_mutex> lock(rw);
    std::cout << "reader " << id << " sees " << data_value << "\n";
}

void writer(int v) {
    std::unique_lock<std::shared_mutex> lock(rw);
    data_value = v;
    std::cout << "writer sets " << data_value << "\n";
}

int main() {
    std::vector<std::thread> threads;

    for (int i = 0; i < 3; ++i) {
        threads.emplace_back(reader, i);
    }
    threads.emplace_back(writer, 42);
    for (int i = 3; i < 6; ++i) {
        threads.emplace_back(reader, i);
    }

    for (auto& t : threads) {
        t.join();
    }
    return 0;
}
```

## 何时适合读写锁

适合：

- 配置缓存、路由表、字典索引等读多写少数据结构。
- 单次写入成本高，但写入频率低。

不适合：

- 写入频繁场景。
- 临界区很短且线程数不高（普通 mutex 可能更简单更快）。

---

## 常见风险

1. 写饥饿：读线程持续进入，写线程长时间拿不到锁。
2. 锁升级问题：先拿读锁再升级写锁容易引发复杂等待。
3. 临界区过大：锁优势被业务耗时抵消。

---

## 优化建议

1. 保持写临界区尽量短。
2. 避免在持锁期间做 I/O。
3. 基准测试后再决定是否使用 shared_mutex。
4. 写多于读时，直接回到普通 mutex。
